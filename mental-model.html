plaid-importer

A self-hosted web app that imports bank transactions (via Plaid) into one or more Actual Budget servers.

Node 22Fastify SQLite (better-sqlite3)eta templates @actual-app/apiPlaid SDK AGPL-3.0

Server-rendered, single long-running container, one SQLite file for state. No SPA.

What it does

Users link banks through Plaid, define profiles (each pointing at one Actual budget), map a bank account to an Actual account within a profile, and sync — manually or on a schedule. A single bank account can map into several profiles, and one Plaid pull fans out to all of them.

Auth & multi-user

  • Accounts live in a users table (bcrypt), with roles admin / member.
  • Login issues an HttpOnly, SameSite=Lax session cookie carrying user_id.
  • Registration at /register is gated by a secret an admin sets under /settings. First-user bootstrap: while zero users exist, registration is open and the first registrant becomes admin.
  • Env seed: on first boot, APP_USER/APP_PASSWORD seed the admin so an existing single-user deployment keeps working. After that they are ignored.
  • Every user's Plaid connections, profiles, mappings, and history are private to them (owner-scoped queries). Trusted-multi-user threat model.

Core model: connections, profiles, mappings

User
 ├─ Plaid Items (connections) ──< Plaid Accounts        the "source"
 └─ Profiles (one per Actual budget) ──< profile_account_mappings
        (server URL, budget id, encrypted secrets)        (profile + account)
                                                          → actual_account_id
                                                          → pending_visible

The mapping key is (profile_id, plaid_account_id), so the same account can feed many budgets, each with its own target account and "show pending" setting.

You can re-open Plaid Link on an existing connection to add or drop which accounts it shares. Dropped accounts are marked Not syncing — they stop syncing but stay visible and keep their mappings. Because Plaid may hand back a new account_id for a re-added account, accounts are reconciled by their stable persistent_account_id (or, failing that, a name/mask/type match) so mappings survive the swap instead of duplicating.

The sync engine: PULL once, DRAIN to many

Plaid's /transactions/sync is a consume-once stream. To fan out to many budgets without paying per budget, each connection is pulled once into a durable local journal; profiles then deliver from the journal independently.

PULL  (per connection, cost = number of connections)
  syncItem(cursor) → delta
  in ONE db txn: append events to plaid_txn_events (payload encrypted)
                 + advance the item cursor

DRAIN (per profile, local — no extra Plaid cost)
  for each profile connected to the pulled connection:
    read journal events > its watermark, filtered to its mapped accounts
    apply pending-promotion / import / removal lifecycle into its budget
    advance watermark on success (down budget simply retries next run)

PRUNE  delete events delivered to every active connected profile
  • Per-(profile, connection) watermark tracks delivery; a late-joining profile starts at the journal head (future transactions only).
  • Idempotent on Actual's imported_id; replays are safe.
  • The Actual API is a process singleton, so drains run sequentially and a run-level lock prevents overlap (manual vs scheduled).

Scheduling

  • A schedule targets a set of connections (Plaid items) and a day-of-week cadence — days_of_week, time_of_day, repeat_weeks (1=weekly, 2=bi-weekly), and a per-schedule timezone. An in-process runner fires due schedules and records them as triggered_by="scheduled" (badged in History).
  • next_run_at is computed (DST-safe) as the next matching weekday/time and persisted so schedules survive restarts. Legacy rows predating this model keep a non-null interval_hours (days_of_week IS NULL) and advance by that interval until edited; editing one migrates it to the new cadence.
  • Because execution is the normal fan-out, syncing a connection's account updates it in every profile that maps it, from a single pull.
  • UI mirrors the Profiles pattern: /schedules is a calm list with a header New schedule button leading to a dedicated form page (/schedules/new, reusing the same cadence form partial as edit). Each row carries an Active/Paused status badge, disabled rows are dimmed, and enable/disable is a fa-toggle-on/fa-toggle-off icon button alongside edit/delete. Legacy rows keep their warning + edit affordance.

Per-connection sync limit

An admin can cap Plaid usage: each connection may be synced at most N times per X hours (set under /settings, disabled by default). Over-limit connections are skipped and the rest still sync; counting is per connection, so batching one-by-one is not penalized.

Known exposure

To write into a budget the app must hold a plaintext copy locally during a sync (the Actual client needs it). Each profile uses its own cache dir which is wiped after the drain, shrinking the plaintext window to "only during an active sync."

SSRF guard

Profile server URLs must be https. Blocking private/loopback hosts is opt-in via BLOCK_PRIVATE_ACTUAL_HOSTS (off by default, since a self-hosted Actual server is normally on a LAN address).

App banners

A server-defined set of active banner keys renders between the topbar and main content on every authenticated page. A banner shows until the user dismisses it (POST /banners/:key/dismiss → row in dismissed_banners), and dismissal is one-and-done across sessions and devices. The first banner (schedule_migration_v1) appears only to users who still hold a legacy schedule.

Internationalization

All user-facing strings come from a catalog (en, es) with English fallback; locale is resolved from Accept-Language. Server templates and client JS both read it.

Deployment (overview)

  • Built as a container; pushed to a registry; run with /app/data mounted for the SQLite DB and Actual cache. Reverse-proxy terminates TLS so the public URL is HTTPS.
  • SQL migrations run at boot (idempotent); a boot-time seed creates the admin and a "Default" profile from env on first upgrade.
  • Config is environment-driven; see .env.example. Deploy specifics live in DEPLOY.md.
